## 1. 目的

本文書は、KKBD-USB 要件定義書（KKBD-USB-REQ-001）に基づき、Raspberry Pi Pico（RP2040）を用いた USB キーボード → UART 変換ファームウェアを段階的に実装するための計画を示す。

各フェーズで動作確認を実施しながら積み上げ型で開発することで、問題の早期発見と手戻りの最小化を図る。

---

## 2. 実装方針

| 方針 | 内容 |
|------|------|
| 段階的開発 | フェーズ 1〜8 に分割し、各フェーズ完了後に動作確認を行ってから次フェーズに進む |
| 受け入れ条件の明確化 | 各フェーズに明確な受け入れ条件（完了判定基準）を設け、曖昧な完了を避ける |
| SDK・ライブラリ活用 | Pico SDK および TinyUSB の標準 API を最大限活用し、独自実装を最小化する |
| ファイル分割 | 機能単位でソースファイルを分割し、可読性・保守性を確保する |
| Pico SDK バージョン固定 | Pico SDK v1.5.1 を使用。TinyUSB バージョンの変動によるビルド失敗を防ぐ |

| Phase 1 | ビルド環境構築 | NFR-001 |

| 項目 | 内容 |

#### 対象ファイル

- [ ] `tuh_hid_report_received_cb()` を実装し、HID レポートを受信して処理する

#### 受け入れ条件

| # | 条件 |
|---|------|
| 1 | USB キーボードで A〜Z キーを押すと、小文字 a〜z が UART に送信される |
| 2 | 0〜9 キーを押すと、数字 0〜9 が UART に送信される |
| 3 | 未対応キー（F キー、矢印キー等）を押しても UART への誤送信が発生しない |
| 4 | キーを押し続けても、離して再度押さない限り同じ文字が繰り返し送信されない（リピートは Phase 6 で対応） |
| 5 | 同じキーを押し続けた場合の差分検出ロジックが正しく動作する（連投が発生しない） |

## 4. テスト戦略

> **詳細**: 拡充したテスト戦略は [`テスト戦略.md`](./テスト戦略.md) を参照。本セクションは概要のみ記載。

### 4.1 テスト種別と方針

| テスト種別 | 対象 | 方法 | 実施タイミング |
|-----------|------|------|--------------|
| ユニットテスト | `keymap.c` の変換テーブル | ホスト PC（x86/arm）でビルド・実行（`stdio.h` / `assert.h` のみ使用） | Phase 4/5 実装中 |
| 結合テスト | ジャンパー読取 + UART 送信 | PC のシリアル端末でボーレート・文字列確認 | Phase 2 完了後 |
| 結合テスト | USB ホスト + キー入力 + UART 送信 | PC のシリアル端末で実際のキー入力確認 | Phase 4/5/6 完了後 |
| システムテスト | 全機能 + SBC 接続 | SBC6800/KZ80 への実接続・操作確認 | Phase 8 |

### 4.2 ユニットテスト詳細

> **詳細**: [`テスト戦略.md`](./テスト戦略.md) の「3. L1: ホストユニットテスト」を参照。

`keymap.c` は Pico SDK / TinyUSB に依存しない実装とし、ホスト PC でコンパイル・実行可能なテストを作成する。

```
tests/
  test_keymap.c   # EXPECT_* マクロベースのユニットテスト
```

**Phase 別テスト有効化**:

| Phase | テスト有効化状況 |
|-------|----------------|
| Phase 1〜3 完了時 | `test_modifier_constants` のみ有効（`MOD_*` 定数値検証） |
| Phase 4 完了後 | `KKBD_PHASE4_DONE` を定義してビルドすることで全テスト有効化 |

Phase 4 完了後の有効化方法:
```bash
cmake -S tests -B build-tests -DKKBD_PHASE4_DONE=ON
cmake --build build-tests
ctest --test-dir build-tests --output-on-failure
```

### 4.3 デバッグ用 UART ログ方針

- デバッグビルド（`CMAKE_BUILD_TYPE=Debug`）では UART ログを有効化
- リリースビルド（`Release`）ではログを無効化し、UART 帯域をキー送信のみに使用
- ログ有効/無効は `tusb_config.h` 内の `#define KKBD_DEBUG 1` で切り替え

## 6. リスクと対策

| # | リスク | 影響度 | 発生可能性 | 対策 |
|---|--------|--------|-----------|------|
| R1 | Pico Micro-B をホストとして使用する際の VBUS 給電回路未検討 | 高（USB 認識失敗） | 中 | VBUS 給電回路の設計を回路図作成フェーズで確認。参考実装（USB 簡単ホスト）を参照 |
| R2 | SBC 側 UART のロジックレベル不一致（RS-232C 等） | 高（通信不能） | 中 | レベル変換 IC（MAX232 等）の使用を検討。接続前に SBC の UART 仕様を確認 |
| R3 | Pico SDK / TinyUSB のバージョン変更による API 非互換 | 中（ビルド失敗） | 低 | Pico SDK v1.5.1 を `pico_sdk_import.cmake` に固定。CI/CD 環境でもバージョンを固定 |
| R4 | キーボード固有の挙動による一部キーの誤認識 | 低〜中 | 中 | Phase 8 で複数キーボード（最低 2 種類）によるテストを実施 |
| R5 | キーリピート実装の精度（タイマー依存） | 低 | 低 | `time_us_64()` / `to_ms_since_boot()` を使用し、精度の高いタイマーを利用 |
| R6 | 24 時間連続動作中のメモリリーク・ハングアップ | 高（信頼性） | 低 | ウォッチドッグタイマー（`watchdog_enable()`）の使用を検討（NFR-004 対応） |
| R7 | USB Hub 経由のキーボード非対応（`CFG_TUH_HUB=0`） | 中（一部キーボード利用不可） | 低 | USB Hub 非対応であることをドキュメントに明記。直接接続を前提とする。将来対応は `CFG_TUH_HUB=1` で拡張可能 |
