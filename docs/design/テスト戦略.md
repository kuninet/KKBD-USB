## 1. はじめに

### 1.1 目的

本文書は、KKBD-USB プロジェクトにおける実装計画（KKBD-USB-IMP-001）と並走するテスト体系を定義する。各フェーズの実装に対して、どのレベルでどのようにテストを実施するかを明確にし、品質を継続的に確保するための指針を示す。

### 1.2 スコープ

本文書が対象とするテストは以下の4レベルとする。

| レベル | 名称 | 対象 |
|--------|------|------|
| L1 | ホストユニットテスト | 純粋ロジック関数（Pico SDK 非依存） |
| L2 | ホスト結合テスト | モジュール間連携（スタブを使用） |
| L3 | シミュレーション | USB・UART 含む統合動作（Wokwi 等） |
| L4 | 実機テスト | エンドツーエンド（Pico 実機 + SBC） |

## 2. テストレベル定義

KKBD-USB ファームウェアのテストは、以下の4レベルで構成する。各レベルは目的・実行環境・自動化の度合いが異なり、それぞれが補完的に機能する。

| レベル | 名称 | 実行環境 | 主目的 | 自動化 |
|--------|------|---------|--------|--------|
| L1 | ホストユニットテスト | 開発PC（gcc/clang） | 純粋ロジックの単体検証 | 完全自動 |
| L2 | ホスト結合テスト | 開発PC + モック | モジュール間連携検証 | 完全自動 |
| L3 | シミュレーション | Wokwi 等 | USB + UART 含む統合動作 | 半自動 |
| L4 | 実機テスト | Pico 実機 + SBC | エンドツーエンド | 手動 |

### 各レベルの位置づけ

```mermaid
graph LR
    L1["L1: ホストユニット<br/>（完全自動・高速）"] --> L2["L2: ホスト結合<br/>（完全自動・中速）"]
    L2 --> L3["L3: シミュレーション<br/>（半自動・低速）"]
    L3 --> L4["L4: 実機テスト<br/>（手動・最終確認）"]
    style L1 fill:#d4edda
    style L2 fill:#d4edda
    style L3 fill:#fff3cd
    style L4 fill:#f8d7da
```

- **L1・L2** はコードコミットのたびに自動実行し、早期に問題を検出する
- **L3** はフェーズ完了時や主要機能実装後に実施する
- **L4** は Phase 8 実機検証フェーズで集中的に実施する

## 3. L1: ホストユニットテスト（主軸）

### 3.1 対象モジュール

Pico SDK に依存しない純粋ロジック関数を対象として、ホスト PC 上でビルド・実行するユニットテストを実施する。

| モジュール | テスト対象 | Pico SDK 依存切り離し |
|-----------|----------|----------------------|
| keymap | `keymap_convert()`, `keymap_is_enter()`, `MOD_*` 定数値 | 不要（純粋関数・SDK 非依存） |

### 3.2 フレームワーク選定

| 項目 | 内容 |
|------|------|
| 採用 | `assert.h` ベースの軽量自前マクロ（`test_framework.h`） |
| 採用理由 | 外部依存ゼロ、CI 設定簡素化、組み込み C プロジェクトとの親和性 |
| 将来移行余地 | Unity / CMock など組み込み C 向けフレームワークへの移行は将来検討 |

#### マクロ仕様（`test_framework.h`）

| マクロ | 用途 |
|--------|------|
| `TEST(name)` | テスト関数宣言 |
| `EXPECT_EQ(expected, actual)` | 値の等値確認（不一致時はメッセージを出力して失敗カウント・処理継続） |
| `EXPECT_TRUE(expr)` | 真値確認（失敗しても処理継続） |
| `EXPECT_FALSE(expr)` | 偽値確認（失敗しても処理継続） |
| `RUN_TEST(func)` | テスト実行・カウントアップ |
| テスト終了 | パス数・失敗数を stdout に出力し、失敗があれば `exit(1)` |

> 注意: `EXPECT_*` は失敗後も処理を継続する（Googletest の EXPECT_* 慣例に従う）。
> 失敗後の続行でクラッシュしうる場合は、呼び出し側で `if (g_current_failed) return;` のガードを入れること。
> 引数順は `(expected, actual)` の標準順とする。

### 3.3 ビルド方法

```bash
# テスト専用 CMake ディレクトリからビルド（src/ のPico SDK設定と独立）
cmake -S tests -B build-tests
cmake --build build-tests
ctest --test-dir build-tests --output-on-failure
```

> 注意: `tests/CMakeLists.txt` はホスト PC 向けのビルドであり、`pico_sdk_import.cmake` を読み込まない。標準の `gcc` / `clang` でコンパイルする。

### 3.4 ディレクトリ構成

## 4. L2: ホスト結合テスト

### 4.1 目的

`usb_host` コールバックから `keymap` → `uart_out` までの連携を、Pico SDK スタブを使用してホスト PC 上で検証する。

### 4.2 方針

| 項目 | 内容 |
|------|------|
| 対象 | モジュール間のデータフロー検証 |
| スタブ対象 | `gpio_get()`, `uart_putc()`, `uart_init()`, `uart_set_baudrate()`, `time_us_64()` |
| スタブ機能 | 呼び出し記録・戻り値の注入・呼び出し回数アサーション |
| 配置 | `tests/integration/` 配下 |
| 実装タイミング | **本 Issue では未実装。Phase 4〜6 完了後に別 Issue で追加** |

### 4.3 スタブ例

```c
/* tests/integration/stubs/pico_stub.h */
/* gpio_get スタブ: テスト側から戻り値を注入可能にする */
static uint32_t stub_gpio_get_return[32];
static int      stub_gpio_get_call_count[32];

uint32_t gpio_get(uint32_t gpio) {
    stub_gpio_get_call_count[gpio]++;
    return stub_gpio_get_return[gpio];
}
```

### 4.4 テストシナリオ（Phase 4-6 完了後）

| シナリオ | 検証内容 |
|---------|---------|
| 通常キー → UART 送信 | HID レポート受信から `uart_putc()` 呼び出しまでの連携 |
| Enter → 行末コード選択 | `keymap_is_enter()` → `uart_out_send_line_ending()` の連携 |
| ジャンパー設定 → ボーレート | GPIO スタブ値 → `config_get_baudrate()` → `uart_init()` 呼び出し |
| キーダウン継続 → リピート | `keyrepeat_register()` → `keyrepeat_task()` → `uart_putc()` タイミング |
| 非 Keyboard デバイス接続 | `tuh_hid_mount_cb()` でプロトコル非一致 → `uart_putc()` 未呼び出し |

## 5. L3: シミュレーション（Wokwi）

### 5.1 概要

Wokwi は RP2040 + USB HID キーボード + UART 出力を仮想化できるオンライン / CLI シミュレータである。実機を用意せずに USB ホスト機能を含む統合動作の一部を検証できる。

### 5.2 セットアップ手順

| # | 手順 |
|---|------|
| 1 | Wokwi CLI のインストール: `npm install -g wokwi-cli` または [公式ドキュメント](https://docs.wokwi.com/wokwi-ci/cli) 参照 |
| 2 | Wokwi アカウントトークン取得（CLI 実行に必要） |
| 3 | プロジェクトに `wokwi.toml` と `diagram.json` を配置（**本 Issue では未作成、別 Issue で対応**） |
| 4 | ビルドした `.elf` ファイルを指定して実行: `wokwi-cli --elf build/src/kkbd_usb.elf` |
| 5 | 仮想 USB キーボードからキー入力 → UART モニタで出力確認 |

> 注意: Wokwi の RP2040 USB Host サポートの完全性については制限がある場合がある。L4 実機テストで補完すること。

### 5.3 テストシナリオ（Phase 4 以降に有効）

| シナリオ | 検証内容 | 有効フェーズ |
|---------|---------|------------|
| 英数字キー入力 | 各 HID Usage ID に対する UART 出力の確認 | Phase 4 完了後 |
| Shift / Ctrl 組み合わせ | 修飾キー付き入力の変換確認 | Phase 5 完了後 |
| Enter 行末コード（3 種類） | JP1/JP2 設定パターン × Enter 押下 | Phase 6 完了後 |
| 4 種ボーレート送信確認 | JP3/JP4 設定パターン × 送信確認 | Phase 6 完了後 |
| USB キーボード切断・再接続 | 切断後に再接続して入力が再開されること | Phase 7 完了後 |
| 非対応デバイス（マウス等） | UART への誤送信が発生しないこと | Phase 7 完了後 |

## 6. L4: 実機テスト（PicoProbe + SBC）

### 6.1 機材

| 機材 | 役割 |
|------|------|
| Raspberry Pi Pico × 2 | DUT（テスト対象）+ PicoProbe（デバッグ用） |
| USB キーボード 複数種類 | 基本動作確認・非対応デバイス異常系確認（FR-009 含む） |
| ロジックアナライザ または USB-UART 変換アダプタ | UART 波形確認・受信文字確認 |
| 接続先 SBC | SBC6800、KZ80 等の実機（エンドツーエンド確認） |
| PC（minicom / TeraTerm 等） | UART 受信確認用シリアル端末 |

### 6.2 書き込み・デバッグ手順

| 方法 | コマンド / 手順 |
|------|----------------|
| USB 経由書き込み | BOOTSEL ボタン押しながら USB 接続 → `picotool load -fx kkbd_usb.uf2` |
| PicoProbe + OpenOCD | SWD 経由書き込み: `openocd -f interface/cmsis-dap.cfg -f target/rp2040.cfg -c "program kkbd_usb.elf verify reset exit"` |
| UART 受信確認（Linux） | `minicom -D /dev/ttyUSB0 -b 9600`（デバイスパスは環境により異なる） |
| UART 受信確認（Mac） | `minicom -D /dev/tty.usbserial-XXXX -b 9600`（XXXX はアダプタのシリアル番号） |
| UART 受信確認（Windows） | TeraTerm でシリアル接続（COM ポートを選択、ボーレートをジャンパー設定に合わせる） |

| keymap Ctrl テーブル | Ctrl + A〜Z（0x01〜0x1A）+ 記号 4 種 | 30 件 |

| 行末コード（L2 対象） | CR / LF / CRLF / 予約=CR デフォルト動作 | 4 件 |
| ボーレート選択（L2 対象） | 9600 / 19200 / 38400 / 115200 | 4 件 |
| エラー処理（FR-009）（L4 対象） | 全 5 異常系（IT-018〜IT-021 含む） | 5 件 |

### 目標カバレッジ方針

| レベル | カバレッジ目標 |
|--------|--------------|
| L1（keymap 通常テーブル・Shift・Ctrl） | 対象 HID Usage ID の **100%** を網羅 |
| L1（`keymap_is_enter()`） | Enter / 非 Enter の全パターン確認 |
| L2（config・uart_out・keyrepeat） | 主要パス **80%** 以上 |
| L4（実機試験項目） | IT-001〜IT-030 の全 30 項目を実施 |

      - uses: actions/checkout@v4

## 11. リスクと対策

| # | リスク | 影響 | 対策 |
|---|--------|------|------|
| R1 | `keymap.c` が実装時に Pico SDK 依存（`pico/stdlib.h` 等）を取り込む | ホストビルド失敗・L1 実行不能 | 設計段階で依存を分離。`keymap.c` には `<stdint.h>` / `<stdbool.h>` のみ使用する |
| R2 | Wokwi の RP2040 USB Host サポートが不完全 | L3 で USB 機能が検証できない | L4 実機での補完試験を充実させる。L3 はベストエフォートとする |
| R3 | ホスト側テストの偽合格（実機挙動との乖離） | L1 合格でも実機で動作しない場合がある | L1 は純粋ロジックのみ検証。ハードウェア依存部分は L4 で必ず実機検証する |
| R4 | スタブの不完全さによる L2 の信頼性低下 | モジュール連携バグを見逃す | スタブの呼び出し記録・回数アサーションを徹底する |
| R5 | `time_us_64()` の引数化忘れによる L1 でのキーリピートテスト困難 | `keyrepeat_task()` が L1 で検証できない | 設計書に記載の通り `time_us_64()` を引数化した設計で実装すること |
| R6 | キーボード機種による HID レポート差異 | 特定キーボードで変換テーブルが機能しない | L4 で複数キーボード（最低 2 種類）を使用して検証する（IT-001〜006） |
